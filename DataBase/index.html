데이터베이스 소개

데이터베이스(Database, DB)는 구조화된 데이터의 조직화된 모음으로, 효율적인 저장, 관리, 검색을 위해 설계되었습니다. 데이터베이스 관리 시스템(DBMS)은 데이터베이스를 생성, 유지, 사용할 수 있는 소프트웨어 시스템입니다.

데이터베이스의 주요 목적:

  • 데이터 중복 최소화
  • 데이터 일관성 및 무결성 유지
  • 효율적인 데이터 검색 및 조작
  • 데이터 보안 강화
  • 데이터 독립성 제공

관계형 데이터베이스

관계형 데이터베이스는 테이블(릴레이션)을 사용해 데이터를 저장하며, 각 테이블은 행(레코드)과 열(필드)로 구성됩니다. 이러한 테이블들은 서로 관계를 가지며 이를 통해 데이터의 무결성을 유지합니다.

관계형 데이터베이스의 주요 구성 요소

테이블(Table)

데이터를 행과 열로 구조화하여 저장하는 기본 단위

행(Row)

레코드 또는 튜플이라고도 하며, 하나의 데이터 항목을 나타냄

열(Column)

속성 또는 필드라고도 하며, 데이터의 특정 유형을 정의

기본 키(Primary Key)

각 행을 고유하게 식별하는 열 또는 열의 조합

외래 키(Foreign Key)

다른 테이블의 기본 키를 참조하는 열로, 테이블 간 관계를 형성

인덱스(Index)

데이터 검색 속도를 향상시키는 데이터 구조

제약 조건(Constraint)

데이터 무결성을 유지하기 위한 규칙

관계 유형

일대일(1:1) 관계

한 테이블의 레코드가 다른 테이블의 하나의 레코드와만 연결

일대다(1:N) 관계

한 테이블의 레코드가 다른 테이블의 여러 레코드와 연결

다대다(N:M) 관계

양쪽 테이블의 레코드가 서로 여러 레코드와 연결 (중간 테이블로 구현)

SQL 기초

SQL(Structured Query Language)은 관계형 데이터베이스 관리 시스템(RDBMS)에서 데이터를 관리하기 위한 표준 언어입니다. SQL은 데이터 정의(DDL), 데이터 조작(DML), 데이터 제어(DCL)로 나눌 수 있습니다.

SQL 명령어 유형

DDL(Data Definition Language): 데이터베이스 구조 정의

  • CREATE: 테이블, 인덱스 등 객체 생성
  • ALTER: 객체 구조 변경
  • DROP: 객체 삭제
  • TRUNCATE: 테이블 데이터 모두 삭제

DML(Data Manipulation Language): 데이터 조작

  • SELECT: 데이터 조회
  • INSERT: 데이터 삽입
  • UPDATE: 데이터 수정
  • DELETE: 데이터 삭제

DCL(Data Control Language): 데이터 접근 제어

  • GRANT: 권한 부여
  • REVOKE: 권한 회수

TCL(Transaction Control Language): 트랜잭션 제어

  • COMMIT: 트랜잭션 확정
  • ROLLBACK: 트랜잭션 취소
  • SAVEPOINT: 트랜잭션 내 저장점 설정

FROM Products

-- 테이블 생성
CREATE TABLE Customers (

    email VARCHAR(100),

CREATE TABLE Orders (
    order_id INT PRIMARY KEY,
    customer_id INT,
    order_date DATE,
    total_amount DECIMAL(10, 2),
    FOREIGN KEY (customer_id) REFERENCES Customers(customer_id)
);

INSERT INTO Customers VALUES (103, '이지은', 'lee@example.com', '인천');
INSERT INTO Customers VALUES (104, '박민지', 'park@example.com', '대구');

-- INNER JOIN: 두 테이블에서 일치하는 레코드만 반환
SELECT o.order_id, c.customer_name, o.order_date, o.total_amount
FROM Orders o
INNER JOIN Customers c ON o.customer_id = c.customer_id;

-- LEFT JOIN: 왼쪽 테이블의 모든 레코드와 오른쪽 테이블의 일치하는 레코드 반환
SELECT c.customer_id, c.customer_name, o.order_id, o.order_date
FROM Customers c
LEFT JOIN Orders o ON c.customer_id = o.customer_id;

-- RIGHT JOIN: 오른쪽 테이블의 모든 레코드와 왼쪽 테이블의 일치하는 레코드 반환
SELECT c.customer_name, o.order_id, o.order_date, o.total_amount
FROM Orders o
RIGHT JOIN Customers c ON o.customer_id = c.customer_id;

-- 주문한 고객의 총 주문 금액
SELECT c.customer_id, c.customer_name, SUM(o.total_amount) as total_spent
FROM Customers c
JOIN Orders o ON c.customer_id = o.customer_id
GROUP BY c.customer_id, c.customer_name
ORDER BY total_spent DESC;

데이터베이스 설계

데이터베이스 설계는 사용자 요구사항을 충족하는 데이터베이스 구조를 개발하는 과정입니다. 효율적이고 효과적인 데이터 저장, 검색, 관리를 보장하는 것이 목표입니다.

데이터베이스 설계 단계

요구사항 분석

사용자 요구사항 파악 및 문서화

개념적 설계

ER 다이어그램 같은 고수준 데이터 모델 생성

논리적 설계

개념적 모델을 관계형 스키마로 변환

물리적 설계

저장소 구조, 인덱스, 접근 경로 등 결정

구현

DDL과 DML을 사용해 데이터베이스 생성 및 초기 데이터 로드

ER(Entity-Relationship) 다이어그램

ER 다이어그램은 개체(엔티티), 속성, 관계를 시각적으로 표현하는 도구입니다.

개체(Entity)

실세계의 객체나 개념을 나타냄 (예: 학생, 강좌)

속성(Attribute)

개체의 특성이나 성질 (예: 이름, 학번)

관계(Relationship)

개체 간의 연관성 (예: 학생이 강좌를 수강한다)

-- 학생 테이블
CREATE TABLE Students (
    student_id INT PRIMARY KEY,
    student_name VARCHAR(100) NOT NULL,
    email VARCHAR(100) UNIQUE,
    major VARCHAR(50),
    year INT
);

-- 강좌 테이블
CREATE TABLE Courses (
    course_id VARCHAR(10) PRIMARY KEY,
    course_name VARCHAR(100) NOT NULL,
    instructor VARCHAR(100),
    credits INT,
    department VARCHAR(50)
);

-- 수강 관계 테이블 (다대다 관계 구현)
CREATE TABLE Enrollments (
    enrollment_id INT PRIMARY KEY AUTO_INCREMENT,
    student_id INT,
    course_id VARCHAR(10),
    semester VARCHAR(10),
    grade VARCHAR(5),
    FOREIGN KEY (student_id) REFERENCES Students(student_id),
    FOREIGN KEY (course_id) REFERENCES Courses(course_id),
    UNIQUE (student_id, course_id, semester)
);

-- 학생별 수강 과목 조회
SELECT s.student_id, s.student_name, c.course_id, c.course_name, e.grade
FROM Students s
JOIN Enrollments e ON s.student_id = e.student_id
JOIN Courses c ON e.course_id = c.course_id
ORDER BY s.student_id, c.course_id;

SELECT c.course_id, c.course_name, COUNT(e.student_id) as student_count
FROM Courses c
LEFT JOIN Enrollments e ON c.course_id = e.course_id
GROUP BY c.course_id, c.course_name;

정규화

정규화(Normalization)는 데이터베이스 설계에서 중복을 최소화하고 데이터 무결성을 향상시키기 위해 관계형 데이터베이스의 테이블을 구조화하는 과정입니다.

정규화의 목적

  • 데이터 중복 최소화
  • 데이터 무결성 유지
  • 효율적인 데이터 구조화
  • 데이터 갱신 이상(anomaly) 방지

정규화 단계

제1정규형(1NF)

모든 속성은 원자적(atomic)이어야 함

제2정규형(2NF)

1NF + 모든 비주요 속성이 기본 키에 완전 함수적 종속

제3정규형(3NF)

2NF + 이행적 종속 제거

BCNF(Boyce-Codd 정규형)

3NF의 강화된 버전

제4정규형(4NF)

BCNF + 다치 종속성 제거

제5정규형(5NF)

조인 종속성 제거

    product_price DECIMAL(10, 2)
);

-- 주문 테이블 (3NF)
CREATE TABLE Orders (
    order_id INT PRIMARY KEY,
    customer_id INT,
    order_date DATE,
    FOREIGN KEY (customer_id) REFERENCES Customers(customer_id)
);

-- 주문 상세 테이블 (3NF)
CREATE TABLE OrderDetails (
    order_id INT,
    product_id VARCHAR(10),
    quantity INT,
    PRIMARY KEY (order_id, product_id),
    FOREIGN KEY (order_id) REFERENCES Orders(order_id),
    FOREIGN KEY (product_id) REFERENCES Products(product_id)
);

-- 원래 데이터 조회 (JOIN 사용)
SELECT o.order_id, c.customer_id, c.customer_name, c.customer_address,
       p.product_id, p.product_name, p.product_price, od.quantity, o.order_date
FROM Orders o
JOIN Customers c ON o.customer_id = c.customer_id
JOIN OrderDetails od ON o.order_id = od.order_id
JOIN Products p ON od.product_id = p.product_id
ORDER BY o.order_id, p.product_id;

인덱스

인덱스(Index)는 데이터베이스 테이블의 검색 속도를 향상시키기 위한 자료구조입니다. 인덱스는 책의 색인과 유사하게, 특정 데이터를 빠르게 찾을 수 있도록 도와줍니다.

인덱스의 장점

  • 검색 속도 향상
  • 정렬된 데이터 검색 효율화
  • GROUP BY 또는 ORDER BY 등 쿼리 성능 개선

인덱스의 단점

  • 추가 저장 공간 필요
  • INSERT, UPDATE, DELETE 작업 시 성능 저하
  • 인덱스 관리 오버헤드

인덱스 유형

단일 열 인덱스

하나의 열에 대한 인덱스

복합 인덱스

두 개 이상의 열을 결합한 인덱스

고유 인덱스

중복 값을 허용하지 않는 인덱스

전체 텍스트 인덱스

텍스트 검색을 위한 인덱스

해시 인덱스

비트맵을 사용한 인덱스

트랜잭션

트랜잭션(Transaction)은 데이터베이스의 상태를 변화시키는 작업의 논리적 단위로, 데이터 무결성을 보장하기 위해 모두 성공하거나 모두 실패해야 하는 일련의 연산을 의미합니다.

트랜잭션의 ACID 속성

원자성(Atomicity)

트랜잭션의 모든 연산이 모두 실행되거나 전혀 실행되지 않아야 함

일관성(Consistency)

트랜잭션 실행 전후의 데이터베이스는 일관된 상태를 유지해야 함

격리성(Isolation)

동시에 실행되는 트랜잭션들이 서로 영향을 미치지 않아야 함

지속성(Durability)

성공적으로 완료된 트랜잭션의 결과는 영구적으로 반영되어야 함

트랜잭션 상태

활동(Active)

트랜잭션이 실행 중인 상태

부분 완료(Partially Committed)

마지막 연산까지 실행했지만 아직 커밋되지 않은 상태

완료(Committed)

트랜잭션이 성공적으로 완료된 상태

실패(Failed)

오류가 발생하여 트랜잭션 실행이 중단된 상태

철회(Aborted)

트랜잭션이 롤백되어 모든 변경사항이 취소된 상태

샘플 테이블: Accounts

account_id account_holder balance
1001 홍길동 1000000
1002 김영희 500000

트랜잭션 처리 예시 (송금): 

-- 계좌 테이블 생성
CREATE TABLE Accounts (
    account_id INT PRIMARY KEY,
    account_holder VARCHAR(100) NOT NULL,
    balance DECIMAL(15, 2) NOT NULL
);

-- 초기 데이터 삽입
INSERT INTO Accounts VALUES (1001, '홍길동', 1000000);
INSERT INTO Accounts VALUES (1002, '김영희', 500000);

-- 트랜잭션을 사용한 송금 예시
START TRANSACTION;

-- 잔액 확인
SELECT balance FROM Accounts WHERE account_id = 1001;

-- 출금 계좌에서 금액 차감
UPDATE Accounts SET balance = balance - 200000 WHERE account_id = 1001;

-- 입금 계좌에 금액 추가
UPDATE Accounts SET balance = balance + 200000 WHERE account_id = 1002;

-- 잔액이 음수가 되었는지 확인
SELECT balance FROM Accounts WHERE account_id = 1001;
SELECT balance FROM Accounts WHERE account_id = 1002;

-- 트랜잭션 완료
COMMIT;

-- 결과 확인
SELECT * FROM Accounts;

-- 트랜잭션 롤백 예시
START TRANSACTION;

-- 출금 계좌에서 과도한 금액 차감 (잔액 부족)
UPDATE Accounts SET balance = balance - 2000000 WHERE account_id = 1001;

-- 입금 계좌에 금액 추가
UPDATE Accounts SET balance = balance + 2000000 WHERE account_id = 1002;

-- 비즈니스 규칙 검증: 잔액이 음수가 되었는지 확인
SELECT balance FROM Accounts WHERE account_id = 1001;

-- 잔액이 음수라면 트랜잭션 롤백
-- 실제로는 프로그래밍 언어에서 조건문을 사용할 것임
ROLLBACK;

-- 결과 확인 (변경사항이 취소됨)
SELECT * FROM Accounts;

동시에 여러 트랜잭션이 실행될 때 발생할 수 있는 문제와 이를 방지하기 위한 격리 수준입니다.

격리 수준 Dirty Read Non-repeatable Read Phantom Read
READ UNCOMMITTED 발생 가능 발생 가능 발생 가능
READ COMMITTED 방지 발생 가능 발생 가능
REPEATABLE READ 방지 방지 발생 가능

트랜잭션 격리 수준 설정: 

-- 현재 세션의 트랜잭션 격리 수준 확인
SELECT @@transaction_isolation;

-- 트랜잭션 격리 수준 설정

동시성 제어

동시성 제어(Concurrency Control)는 여러 트랜잭션이 동시에 실행될 때 데이터 일관성과 무결성을 유지하기 위한 기법입니다.

동시성 문제

분실 수정(Lost Update)

한 트랜잭션의 변경이 다른 트랜잭션에 의해 덮어쓰여지는 문제

더티 리드(Dirty Read)

커밋되지 않은 데이터를 읽는 문제

반복 불가능한 읽기(Non-repeatable Read)

같은 트랜잭션 내에서 같은 데이터를 여러 번 읽을 때 결과가 다른 문제

팬텀 읽기(Phantom Read)

트랜잭션 실행 중 다른 트랜잭션이 새로운 행을 삽입하여 이전에 보이지 않던 데이터가 보이는 문제

동시성 제어 기법

락킹(Locking)

데이터에 대한 접근을 제한하는 기법

  • 공유 락(Shared Lock): 읽기 작업을 위한 락
  • 배타적 락(Exclusive Lock): 쓰기 작업을 위한 락

2단계 락킹(Two-Phase Locking)

확장 단계와 수축 단계로 나누어 락을 관리

타임스탬프 기반(Timestamp-based)

트랜잭션마다 타임스탬프를 부여하여 순서를 정함

낙관적 동시성 제어(Optimistic Concurrency Control)

충돌이 드물다고 가정하고, 커밋 시 충돌 검사

다중 버전 동시성 제어(MVCC)

데이터의 여러 버전을 유지하여 읽기 작업이 쓰기 작업을 차단하지 않도록 함

샘플 테이블: Products

product_id product_name stock price
1 노트북 10 1200000
2 스마트폰 20 800000

락을 사용한 동시성 제어 예시: 

-- 테이블 생성
CREATE TABLE Products (
    product_id INT PRIMARY KEY,
    product_name VARCHAR(100) NOT NULL,
    stock INT NOT NULL,
    price DECIMAL(10, 2) NOT NULL
);

-- 데이터 삽입
INSERT INTO Products VALUES (1, '노트북', 10, 1200000);
INSERT INTO Products VALUES (2, '스마트폰', 20, 800000);

-- 세션 1: 상품 재고 확인 후 구매 (락 사용)
START TRANSACTION;

-- 읽기 작업을 위한 공유 락
SELECT * FROM Products WHERE product_id = 1 LOCK IN SHARE MODE;

-- 재고가 충분하다고 판단되면 구매 진행 (배타적 락으로 전환)
UPDATE Products SET stock = stock - 1 WHERE product_id = 1;

COMMIT;

-- 세션 2: 동시에 같은 상품 구매 시도
START TRANSACTION;

-- 세션 1이 공유 락을 가지고 있어도 읽기 가능
SELECT * FROM Products WHERE product_id = 1;

-- 하지만 세션 1이 배타적 락을 획득하면 업데이트 불가 (대기)
UPDATE Products SET stock = stock - 1 WHERE product_id = 1;

COMMIT;

-- SELECT FOR UPDATE: 읽기와 동시에 배타적 락 획득
START TRANSACTION;

-- 다른 트랜잭션은 이 행에 대한 변경이나 SELECT FOR UPDATE 불가
SELECT * FROM Products WHERE product_id = 1 FOR UPDATE;

-- 재고 감소
UPDATE Products SET stock = stock - 1 WHERE product_id = 1;

COMMIT;

-- 낙관적 동시성 제어의 구현 예시 (버전 컬럼 사용)
CREATE TABLE ProductsWithVersion (
    product_id INT PRIMARY KEY,
    product_name VARCHAR(100) NOT NULL,
    stock INT NOT NULL,
    price DECIMAL(10, 2) NOT NULL,
    version INT NOT NULL DEFAULT 1  -- 버전 정보
);

INSERT INTO ProductsWithVersion VALUES (1, '노트북', 10, 1200000, 1);
INSERT INTO ProductsWithVersion VALUES (2, '스마트폰', 20, 800000, 1);

-- 낙관적 동시성 제어를 사용한 재고 업데이트
START TRANSACTION;

-- 상품 정보와 버전 읽기
SELECT product_id, stock, version FROM ProductsWithVersion WHERE product_id = 1;
-- 결과: product_id=1, stock=10, version=1

-- 재고 감소 및 버전 증가 (동시에 다른 세션이 변경하지 않았다고 가정)
UPDATE ProductsWithVersion 
SET stock = stock - 1, version = version + 1 
WHERE product_id = 1 AND version = 1;

-- 영향 받은 행이 있는지 확인 (0이면 다른 트랜잭션이 이미 변경함)
-- 실제로는 애플리케이션 코드에서 처리

COMMIT;

참고: MySQL의 InnoDB 엔진은 기본적으로 MVCC(Multi-Version Concurrency Control)를 사용하여 동시성을 관리합니다. 이를 통해 읽기 작업이 쓰기 작업을 차단하지 않도록 하면서도 트랜잭션 격리성을 보장합니다.

쿼리 최적화

쿼리 최적화(Query Optimization)는 데이터베이스 쿼리의 성능을 향상시키기 위한 기법입니다. 적절한 쿼리 최적화는 응답 시간을 줄이고 시스템 자원을 효율적으로 사용할 수 있게 합니다.

쿼리 재작성

더 효율적인 형태로 쿼리 재구성

조인 최적화

효율적인 조인 순서 및 방식 선택

서브쿼리 최적화

서브쿼리를 조인으로 변환하거나 최적화

불필요한 컬럼 제거

SELECT * 대신 필요한 컬럼만 명시

필요한 결과만 제한하여 조회

캐싱

자주 사용되는 쿼리 결과 캐싱

대규모 주문 데이터베이스의 테이블 구조:

Customers 테이블

... ... ... ... ...

Orders 테이블

... ... ... ... ...

쿼리 최적화 예시: 

-- 테이블 생성 및 인덱스 설정
CREATE TABLE Customers (
    customer_id INT PRIMARY KEY,
    name VARCHAR(100) NOT NULL,
    email VARCHAR(100) UNIQUE NOT NULL,
    city VARCHAR(50),
    registration_date DATE
);

CREATE TABLE Orders (
    order_id INT PRIMARY KEY,
    customer_id INT NOT NULL,
    order_date DATETIME NOT NULL,
    status VARCHAR(20) NOT NULL,
    total_amount DECIMAL(10, 2) NOT NULL,
    FOREIGN KEY (customer_id) REFERENCES Customers(customer_id),
    INDEX idx_customer_date (customer_id, order_date),
    INDEX idx_date (order_date),
    INDEX idx_status (status)
);

CREATE TABLE Products (
    product_id INT PRIMARY KEY,
    product_name VARCHAR(100) NOT NULL,
    category VARCHAR(50),
    price DECIMAL(10, 2) NOT NULL,
    INDEX idx_category (category)
);

CREATE TABLE OrderItems (
    item_id INT PRIMARY KEY,
    order_id INT NOT NULL,
    product_id INT NOT NULL,
    quantity INT NOT NULL,
    price DECIMAL(10, 2) NOT NULL,
    FOREIGN KEY (order_id) REFERENCES Orders(order_id),
    FOREIGN KEY (product_id) REFERENCES Products(product_id),
    INDEX idx_order (order_id),
    INDEX idx_product (product_id)
);

-- 비효율적인 쿼리 예시 1: SELECT *의 사용
SELECT * FROM Orders WHERE customer_id = 1001;

-- 최적화된 쿼리 1: 필요한 컬럼만 선택
SELECT order_id, order_date, status, total_amount 
FROM Orders WHERE customer_id = 1001;

-- 비효율적인 쿼리 예시 2: IN 대신 JOIN 사용
SELECT * FROM Customers 
WHERE customer_id IN (
    SELECT customer_id FROM Orders WHERE order_date > '2023-01-01'
);

-- 최적화된 쿼리 2: JOIN 사용
SELECT DISTINCT c.* 
FROM Customers c
JOIN Orders o ON c.customer_id = o.customer_id
WHERE o.order_date > '2023-01-01';

-- 비효율적인 쿼리 예시 3: 인덱스를 활용하지 않는 조건
SELECT * FROM Orders 
WHERE YEAR(order_date) = 2023;

-- 최적화된 쿼리 3: 인덱스를 활용할 수 있는 형태로 변경
SELECT * FROM Orders 
WHERE order_date >= '2023-01-01' AND order_date < '2024-01-01';

-- 비효율적인 쿼리 예시 4: 불필요한 ORDER BY
SELECT * FROM Customers 
ORDER BY customer_id 
LIMIT 10;

-- 최적화된 쿼리 4: 기본 키는 이미 정렬되어 있음
SELECT * FROM Customers 
LIMIT 10;

-- 복잡한 쿼리 최적화 (카테고리별, 월별 매출 집계)
-- 비효율적인 방식
SELECT p.category, 
       MONTH(o.order_date) as month, 
       SUM(oi.quantity * oi.price) as total_sales
FROM OrderItems oi
JOIN Orders o ON oi.order_id = o.order_id
JOIN Products p ON oi.product_id = p.product_id
WHERE YEAR(o.order_date) = 2023
GROUP BY p.category, MONTH(o.order_date)
ORDER BY p.category, month;

-- 최적화된 방식
SELECT p.category, 
       MONTH(o.order_date) as month, 
       SUM(oi.quantity * oi.price) as total_sales
FROM Orders o
JOIN OrderItems oi ON o.order_id = oi.order_id
JOIN Products p ON oi.product_id = p.product_id
WHERE o.order_date >= '2023-01-01' AND o.order_date < '2024-01-01'
GROUP BY p.category, MONTH(o.order_date)
ORDER BY p.category, month;

-- 쿼리 실행 계획 확인
EXPLAIN SELECT o.order_id, c.name, o.order_date, o.total_amount
FROM Orders o
JOIN Customers c ON o.customer_id = c.customer_id
WHERE o.order_date > '2023-01-01'
ORDER BY o.order_date DESC
LIMIT 10;

-- 임시 테이블 활용 예시 (큰 데이터셋 처리)
CREATE TEMPORARY TABLE temp_monthly_sales AS
SELECT YEAR(o.order_date) as year,
       MONTH(o.order_date) as month,
       SUM(o.total_amount) as total_sales
FROM Orders o
WHERE o.status = 'completed'
GROUP BY YEAR(o.order_date), MONTH(o.order_date);

-- 임시 테이블 활용
SELECT * FROM temp_monthly_sales
ORDER BY year DESC, month DESC;

-- 임시 테이블 삭제
DROP TEMPORARY TABLE temp_monthly_sales;

참고: 쿼리 최적화는 데이터베이스의 크기, 스키마, 인덱스, 하드웨어 리소스 등에 따라 달라질 수 있습니다. 항상 EXPLAIN을 사용하여 쿼리 실행 계획을 확인하고, 실제 환경에서 성능을 테스트하는 것이 중요합니다.

NoSQL 데이터베이스

NoSQL(Not Only SQL) 데이터베이스는 관계형 데이터베이스의 제약에서 벗어나 유연한 스키마, 수평적 확장성, 특정 데이터 모델에 최적화된 성능을 제공하는 데이터베이스 시스템입니다.

NoSQL 데이터베이스의 특징

유연한 스키마

고정된 테이블 구조 없이 데이터 저장

수평적 확장성

데이터 분산을 통한 확장 용이

대량 데이터 처리

빅데이터 처리에 적합

다양한 데이터 모델

문서, 키-값, 컬럼, 그래프 등

CAP 이론

일관성(Consistency), 가용성(Availability), 분할 내성(Partition tolerance) 중 2가지 선택

NoSQL 데이터베이스 유형

문서형(Document)

JSON, BSON 등의 문서로 데이터 저장 (MongoDB, CouchDB)

키-값형(Key-Value)

단순한 키-값 쌍으로 데이터 저장 (Redis, DynamoDB)

컬럼형(Column-family)

컬럼 단위로 데이터 저장 (Cassandra, HBase)

그래프형(Graph)

노드와 엣지로 관계 표현 (Neo4j, JanusGraph)

분산 데이터베이스

분산 데이터베이스(Distributed Database)는 물리적으로 다른 위치에 있는 여러 컴퓨터 시스템에 데이터를 분산하여 저장하고 관리하는 데이터베이스 시스템입니다. 분산 데이터베이스는 확장성, 가용성, 성능 향상을 위해 설계되었습니다.

분산 데이터베이스의 특징

위치 투명성

사용자가 데이터의 물리적 위치를 알 필요 없음

복제 투명성

데이터 복제가 사용자에게 보이지 않음

장애 투명성

부분적인 시스템 장애에도 서비스 계속 제공

병렬 처리

여러 노드에서 동시에 쿼리 처리

확장성

노드 추가로 시스템 용량 및 성능 확장 가능

분산 데이터베이스 설계 방식

수평 분할(Sharding)

동일한 스키마의 다른 행들을 다른 노드에 분산

수직 분할(Vertical Partitioning)

테이블의 열을 기준으로 분할

복제(Replication)

동일한 데이터를 여러 노드에 복제하여 저장

마스터-슬레이브(Master-Slave)

마스터는 쓰기, 슬레이브는 읽기 담당

다중 마스터(Multi-Master)

여러 노드가 쓰기 연산 수행 가능

-- US 샤드 (서버 1)

USE shard_us;

CREATE TABLE customers (
    customer_id INT PRIMARY KEY,
    name VARCHAR(100) NOT NULL,
    region VARCHAR(10) NOT NULL,
    email VARCHAR(100) UNIQUE NOT NULL,
    address TEXT,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- 5. MySQL 복제(Replication) 설정
-- 마스터 서버 설정 (my.cnf)
/*
[mysqld]
server-id=1
log_bin=mysql-bin
binlog_format=ROW
*/

-- 슬레이브 서버 설정 (my.cnf)
/*
[mysqld]
server-id=2
relay_log=mysql-relay-bin
log_bin=mysql-bin
binlog_format=ROW
read_only=1
*/

-- 마스터 서버에서 복제 사용자 생성
CREATE USER 'repl_user'@'%' IDENTIFIED BY 'password';
GRANT REPLICATION SLAVE ON *.* TO 'repl_user'@'%';
FLUSH PRIVILEGES;

-- 마스터 상태 확인
SHOW MASTER STATUS;
-- 결과: File: mysql-bin.000001, Position: 1234

-- 슬레이브 서버에서 마스터 연결 설정
CHANGE MASTER TO
  MASTER_HOST='master-server',
  MASTER_USER='repl_user',
  MASTER_PASSWORD='password',
  MASTER_LOG_FILE='mysql-bin.000001',
  MASTER_LOG_POS=1234;

-- 슬레이브 시작
START SLAVE;

-- 슬레이브 상태 확인
SHOW SLAVE STATUS\G

-- 서버 설정 (my.cnf)

-- 복제 사용자 생성
CREATE USER 'repl_user'@'%' IDENTIFIED BY 'password';
GRANT REPLICATION SLAVE ON *.* TO 'repl_user'@'%';
FLUSH PRIVILEGES;

-- 복제용 채널 생성
CHANGE MASTER TO MASTER_USER='repl_user', MASTER_PASSWORD='password' 
FOR CHANNEL 'group_replication_recovery';

-- 첫 번째 서버 부트스트랩
SET GLOBAL group_replication_bootstrap_group=ON;
START GROUP_REPLICATION;
SET GLOBAL group_replication_bootstrap_group=OFF;

-- 다른 서버에서 그룹에 조인
START GROUP_REPLICATION;

-- 그룹 상태 확인
SELECT * FROM performance_schema.replication_group_members;

참고: 실제 분산 데이터베이스 구현은 위 예시보다 훨씬 복잡합니다. 데이터 일관성, 트랜잭션 관리, 장애 처리 등 고려해야 할 사항이 많습니다. 대규모 분산 환경에서는 특화된 데이터베이스 시스템(Google Spanner, Amazon Aurora, Vitess 등)을 사용하는 것이 일반적입니다.
